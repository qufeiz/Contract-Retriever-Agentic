## [2026-06-10] incident | Live cross-session upload LEAK — Bash scope check trusted a substring, not per-path
The verifier caught a REAL, shipping-blocking isolation breach on the live demo: on 1 of 3 tries, session B (which uploaded only its own file) **read session A's uploaded CSV** and returned A's private data. Non-deterministic — the model self-refused the other 2 runs, which is the tell: **the boundary was being held by the model's discretion, not the hook.** (My single manual isolation check passed only because it hit a refusal run — a **false green**.) **Root cause:** widening the read-scope from one root (`knowledge/`) to per-session roots exposed a hole in `_bash_is_readonly_kb` — it proved scope with a SUBSTRING check (`"knowledge" in cmd` / "does the command MENTION the session dir?"). With two allowed roots, a command could **name its own session AND read another session's dir** in one call (`cat uploads/<MINE>/x uploads/<OTHER>/secret`, a pandas read of both, even `ls uploads/<OTHER>` via a 'scaffold' program) and be allowed. `_path_in_kb` itself was correct; only the Bash command-level check was loose. **Fix:** `_PATH_TOKEN_RE` extracts EVERY `knowledge/`//`uploads/` path token (quoted/bare/relative/absolute/`/app/…`) and `_all_named_paths_in_scope` requires `_path_in_kb` for EACH — one out-of-scope path denies the whole command (applied to every Bash command incl. scaffolds + python). **Proven 3 ways:** unit `test_hardening.py::test_bash_cannot_read_another_session_even_if_it_names_its_own` (every leak vector denied, legit own-session allowed; removable-handler-proof); real-SDK `backend/eval/upload_run.py::isolation_gate` (N cross-session attempts, no leak); `backend/eval/_deny_enforcement_probe.py` (positive control: a deny-all hook STARVES the agent — 0 `open` steps — proving a `_pre_tool_use` deny truly blocks the tool, not advisory). META-MISS sealed: my prior tests only checked the pure `_path_in_kb`, never the multi-path Bash hole nor that a deny actually blocks. Gotcha: [`upload-scope-substring-check-leaked-other-sessions.md`](gotchas/upload-scope-substring-check-leaked-other-sessions.md). Redeployed + re-verified on Fly.

## [2026-06-10] fix | Async-job transport (G-1 no longer 504s on the public URL) + cheaper 38-row grounding
The known constraint from the deploy entry — G-1 (the 38-row contract table, several minutes) **504-ing through Vercel's hard 300s function cap** — is **resolved** by moving the long run off the request path. New **async-job transport**: `POST /api/ask/jobs` returns a `job_id` immediately, the agent runs in a **Fly background task** (no cap), and the frontend **polls** `GET /api/ask/jobs/{id}` (~1.5s cadence) streaming the **live agent trace** until the final cited answer. The synchronous `/api/ask` is kept for direct-to-Fly / back-compat callers. The job store is in-process (single-machine Fly, same assumption as the rate limiter), finished jobs pruned after `JOB_TTL_SEC`. The agent's read-only **`PreToolUse` hardening hook is unchanged** — the async path reuses `answer_question` verbatim, so the deny/guard matrix still holds. **G-1 grounding/latency:** the prompt now instructs the agent to compute the whole 38-row set in ONE pandas pass and ground it as a **headline aggregate (COUNT+SUM) cited once + a 5-row earliest-expiring cited sample** rather than 30+ per-row citation tokens — fewer tokens, deterministic, and it avoids the wrong-row-key validation failures that hand-typing invited. Locked by [`backend/tests/test_async_jobs.py`](../backend/tests/test_async_jobs.py); docs updated in [`04`](features/agentic-knowledge-assistant/04-implementation.md).
